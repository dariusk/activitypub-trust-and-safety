Publishers may wish to mark certain content as "sensitive" depending on their audience's sensitivities. This typically hides the content behind a content warning, blurs or collapses media attachments, and other such features requiring click-through consent before viewing the content.

This approach currently lacks a formally defined processing model for specifying which properties are considered "sensitive" or "content". This profile aims to define such a profile and provide guidance on how to handle sensitive content for publishers and consumers of Activity Streams 2.0 documents.

For more details on the status of this document, see the following GitHub issues:

  • Explainer: Handling sensitive content in ActivityPub (swicg/activitypub-trust-and-safety#118)
  • Explainer: Content Warnings as they exist today (swicg/activitypub-trust-and-safety#117)

History

As described in [[[miscellany]]]:

The first software packages that implemented [ActivityPub] began to appear in 2018 [...] Some terms were proposed for inclusion in the ActivityPub specification, but weren't included because of time constraints [...] In the absence of a clear extensions process, those terms were not given their own namespace or context document, but instead were defined in the default namespace for Activity Streams 2.0 [...] Newer [vendor-specific] extensions have been defined in [vendors'] own [namespaces]. The first few terms, though, remain undocumented in a formal specification.

[[[miscellany]]], 1. Introduction

Semantics

One such term is `sensitive`, which is defined in [[[miscellany]]] as follows:

The sensitive property (a boolean) on an object indicates that some users may wish to apply discretion about viewing its content, whether due to nudity, violence, or any other likely aspects that viewers may be sensitive to. This is comparable to what is popularly called "NSFW" (Not Safe For Work) or "trigger warning" in some systems. Implementations may choose to hide content flagged with this property by default, exposed at user discretion. Original discussion in [[[[originalIssue]]]].

Processing model

Content Warnings

In Mastodon, the presence of as:sensitive on an object will collapse the `content` and any `attachment` behind a "content warning", whose text is determined by the `summary`.

Which properties are "sensitive"?

On an Object where as:sensitive is set to `true`, the following properties are generally unsafe to show:

  • `content` feels fairly obvious
  • `attachment` may or may not be -- Mastodon says yes
  • `tag` may be sensitive as well but some tags will be safe
  • `icon`?
  • `image`?

and the following properties might be safe to show:

  • `summary`

On a Link where as:sensitive is set to `true`, pretty much all properties are unsafe to show? In particular:

  • `href`
  • `name`
  • `preview` (see also [[[fepLinkPreview]]])
    • `name`
    • `summary`
    • `image`

and it is probably safe to show:

  • `preview`.`type`

Explicitly signaling which properties are sensitive

However, it can be worth defining an explicit signal for which properties are sensitive or not, so that there is no ambiguity on the publishing or consuming side. Potential approaches:

  • A "sensitivePreview" property that includes the safe information
  • A "sensitiveContent" property that includes the unsafe information
  • A "sensitiveProperties" property (`@type: @vocab`) which indicates the exact properties to be hidden, for example: 
    							{
								"@context": "https://www.w3.org/ns/activitystreams",
								"as:sensitive": true,
								"https://vocab.example/sensitiveProperties": [
									{"id": "as:content"},
									{"id": "as:attachment"}
								]
							}
						
    							{
								"@context": [
									{
										"sensitive": "https://www.w3.org/ns/activitystreams#sensitive",
										"sensitiveProperties": {
											"@id": "https://vocab.example/sensitiveProperties",
											"@type": "@vocab"
										}
									},
									"https://www.w3.org/ns/activitystreams"
								],
								"sensitive": true,
								"sensitiveProperties": [
									"content",
									"attachment"
								]
							}
